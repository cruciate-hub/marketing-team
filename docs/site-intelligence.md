# Site Intelligence

Claude skill for querying, auditing, and analyzing the social.plus marketing website using a structured JSON snapshot of all page content.

Gives full awareness of what every page says — so you can ask questions, find gaps, detect contradictions, and generate site-aware recommendations grounded in the exact text from the live site.

## What it does

- Loads per-file JSON snapshots of the site (marketing, industry, use cases, blog, glossary, answers, customer stories, release notes, product updates, webinars) — each auto-regenerated by a Cloudflare Worker on Webflow CMS publish.
- Quotes exact site text in every answer — never paraphrases, never hallucinates.
- Runs in five analysis modes (Query, Consistency Audit, Coverage Map, Page Health Check, Competitive Comparison) — or combines them when a question spans modes.
- Ends every response with a "While looking at this..." section of 2–3 observations the user didn't ask for but would want to know.

## When it triggers

Any question that depends on what's currently on the social.plus website. Trigger phrases include "what do we say about", "which pages mention", "does the site cover", "our website", "our pages", "site content", "messaging consistency", or page-specific questions like "what does the pricing page say" or "do we mention AI on the homepage".

Also trigger when the user is writing new copy and needs awareness of what already exists, or when auditing for contradictions, stale content, or messaging drift.

The skill does **not** cover docs.social.plus, legal pages, the forum, or any page that isn't a published Webflow page in one of the 10 inventories.

### How to choose which files
- **"Does our pricing page mention X?"** → `pages-marketing.json` only.
- **"Have we ever written about X?"** → `pages-blog.json` + `pages-answers.json` + `pages-glossary.json`.
- **"Are our customer stories consistent?"** → `pages-customer-stories.json` only.
- **"Audit the site for term drift"** → all 10 (warn the user this is context-expensive).
- **"What changes shipped last month?"** → `pages-release-notes.json` + `pages-product-updates.json`.
- **When in doubt** — start with marketing + use cases; expand only if needed.

### Scope notice
Every response that discusses "the website" states which file(s) were loaded. "Across the website" is inaccurate if only marketing was loaded — say "across the 22 marketing pages" instead.

## Five analysis modes

### Query
"What do we say about X?" — returns every match with page URL, heading hierarchy, and exact surrounding text. Ends with a "Notable absences" section flagging pages where the topic is relevant but missing.

### Consistency Audit
Scans for contradictions across numerical claims, feature names/descriptions, feature lists vs. counts, positioning statements, CTA consistency, and product scope claims. Findings grouped by severity (🔴 contradictions / 🟡 inconsistencies / 🟢 minor drift) with the exact conflicting quotes and a resolution recommendation.

### Coverage Map
Maps where a theme appears across the site. Every page is categorized as **Strong presence**, **Mentioned**, **Absent but relevant**, or **Not applicable**. Always shows the compact visual map first (bird's-eye view) before the per-page detail. For every "Missing" page, suggests a specific section and angle.

### Page Health Check
Comprehensive check on a specific page: internal consistency, cross-site consistency (does `/chat` align with `/chat/features`?), content completeness vs. sibling pages, messaging alignment with brand guidelines, structural analysis. Starts with a health score + one-line summary, then detailed findings grouped as "working well / needs attention / recommendations".

### Competitive Comparison
Fetches a competitor URL via WebFetch, loads the comparable social.plus pages, analyzes across five dimensions (claims they make we don't / claims we make they don't / head-to-head messaging / social proof / content depth). Ends with 3–5 prioritized actions grounded in brand guidelines.

## Workflow

1. Fetch `brain.md` (main brain).
2. Load the relevant site JSON files based on the question.
3. Load brand guidelines via `messaging/brain.md` if the analysis involves evaluating messaging quality, suggesting copy, or checking brand consistency.
4. Route to the appropriate mode. Combine modes if the question spans them.
5. Always add the "While looking at this..." section with 2–3 specific, quotable, actionable observations.

## General principles

- **Quote, don't paraphrase.** Use exact text from the JSON. The user needs to know what the site *actually* says, not an interpretation.
- **Be specific about locations.** "On `/chat/features`, under the `## Messaging` section, after the paragraph about typing indicators, add…" — not "the features page could mention this".
- **Page-by-page actionability.** Group recommendations by page whenever possible — the person acting works page-by-page in Webflow.
- **Don't over-flag.** Some pages intentionally use different framing for different audiences. Flag genuinely problematic divergence; note intentional variation as such.
- **Combine modes when natural.** "Audit the pricing page and compare it to [competitor]" runs Page Health Check + Competitive Comparison together.

## The "While looking at this..." section

Every response ends with 2–3 observations the user didn't ask for but would want to know. This is what makes the skill feel like a smart colleague, not a search engine.

Good observations:
- Cross-page connections ("`/use-case/live-chat` and `/chat/features` describe typing indicators differently")
- Missing opportunities ("The moderation page doesn't link to `/industry/gaming` despite the obvious fit")
- Patterns the user can act on ("5 of 10 industry pages use identical SDK boilerplate; the ones that perform differently have industry-specific copy")

Bad observations: generic advice, restating the obvious, guesses that require data you don't have.

## URL format

All reference files are loaded from a shallow clone of this repo (`git clone --depth 1`) into `$MT_REPO`. The canonical fetch block at the top of each SKILL.md handles the clone; skills then read files with `cat "$MT_REPO/<path>"`.
