# Claude design/code тЖТ Webflow playbook

Verified on the official Webflow Connector (`mcp.webflow.com`). Depth lives in the reference files тАФ leave them unread until needed:
- `references/pitfalls.md` тАФ full anti-pattern + gotcha catalog (the numbered items indexed below)
- `references/worked-examples.md` тАФ real before/after diffs (incl. the responsive sub-nav, native-Navbar gutter alignment + a full accessibility/semantic-HTML pass)
- `references/variable-ids.md` тАФ social.plus site ID + pre-mapped variable IDs (load ONLY when binding variables on that site)

## Core principle
**Native first, code only for what Webflow can't express.** Native styles are Style-panel-adjustable and publish cleanly; code that duplicates a native rule silently overrides it by cascade order. For the initial build of a whole section, **`data_whtml_builder` in one call** beats rebuilding element-by-element.

## The tool stack тАФ what's headless, what needs the Designer

## MCP 2.0 capability quick-list (easy to miss)
- **Registered scripts** тАФ `data_scripts_tool register_inline_script` + `add_site_script` / `add_page_script`: individually managed scripts per site/page instead of one freeform blob тАФ read Pitfall 43 first (2000-char cap, invisible to the owner, undeletable).
- **Asset compression** тАФ `data_assets_tool compress_assets` тЖТ webp/avif, replaces files in place (confirm first).
- **Branching** тАФ `data_pages_tool create_branch` / `merge_branch` / `publish_branch`: git-style branches for safe iteration (тЖТ Publish).
- **Build from HTML** тАФ `data_whtml_builder`: raw HTML/CSS strings тЖТ native Webflow elements (next section).
- **Agent instructions** тАФ `data_agent_instructions_tool`: site-level markdown rules auto-distributed to every connected agent.
- **Component instances** тАФ `data_component_builder insert_in_element` / `insert_in_slot`: insert instances by component name.
- **Collection-list wiring** тАФ a list's `source` (collectionId), `filters` and `sort` live on the **DynamoWrapper** element: `data_element_settings_tool set_settings` with `static_json` (e.g. `filters` = `[{"fieldSlug":"category-2","operator":"equals","value":"<item-id>"}]`) тАФ repointing a duplicated page's CMS list to a new category is one call.
- **Analytics** тАФ `data_analyze_tool`: traffic, top pages, events (requires the Analyze add-on).

## Fastest build path тАФ `data_whtml_builder`
One call turns `html` + `css` into a native Webflow section (auto-mapped to classes + breakpoints). Constraints (from the Webflow guide):
- **Single root element.** No `<style>` in `html` тАФ CSS goes in the `css` param.
- **No `@keyframes`, no custom `@media`** тАФ Webflow breakpoints only (below). Keep keyframes / custom-media in freeform head code.
- Build the section, THEN refine natively (bind variables, split hybrids). Far fewer tokens than N ├Ч `data_element_builder` + `create_style`.
- **Quirks to fix right after the call (тЖТ Pitfalls 22-24, 46-47):** write combo-class CSS as the **full chained selector** (`.base.combo{тАж}`, never `.combo{тАж}` alone тАФ a bare combo selector spawns an orphan global rule + an *empty* combo record); it **drops `target="_blank"`** and leaves `href`/`rel` as raw attributes тЖТ re-set the link via `data_element_settings_tool set_settings` key `link`, then `remove_attribute`; and it does **not** bind `<img src>` to the asset library even at the canonical CDN URL тЖТ `set_image_asset` by asset ID afterwards. It also **strips a class that is a span's ONLY class** when the `css` param has no rule for it (27 icon spans lost `material-symbols-outlined` in one build; unknown classes stacked after a known class survive, as do `data-*`/`role`/`aria-*`) тАФ give every must-survive class a css-param rule, recovery recipe in Pitfall 46; and text edits on its plain div Blocks need the **String child**, not the Block (Pitfall 47). It DOES keep inline `<svg>` as native DOM, plus `<br>`, soft hyphens and curly quotes verbatim тАФ no HtmlEmbed detour for those.

## Native vs code decision
**Native** (`data_style_tool` Style record): single-class rules; combo classes (`parent_style_names`); supported pseudos (`:hover :focus :focus-visible :active ::before ::after :nth-child(odd/even) :first/last-child :focus-within тАж`); responsive breakpoint values; variable bindings (`variable_as_value`).
**Code** (freeform head/footer, or an HTML embed): `@keyframes`, `@property`; descendant (`.a .b`) + attribute (`[data-x]`) selectors; `:nth-child(N)`; custom / feature queries (`prefers-reduced-motion`); `color-mix()` (Designer corrupts it on save тАФ keep in code, OR store it as a variable `custom_value` and bind the property to that variable); section-root `--var:` definitions descendants inherit; all JS behavior.

## Breakpoints (authoritative)
`main` = base/desktop (тЙе992) ┬╖ `medium` тЙд991 (`max-width:991px`) ┬╖ `small` тЙд767 (`max-width:767px`) ┬╖ `tiny` тЙд479 (`max-width:479px`) ┬╖ up-scaling `large`тЙе1280 / `xl`тЙе1440 / `xxl`тЙе1920. **Styles cascade DOWN** (mainтЖТmediumтЖТsmallтЖТtiny): set at `small` and it also applies at `tiny` unless overridden. In em: 48em=768, 47.9375em=767, 61.9375em=991, 29.9375em=479. Round Claude's odd breakpoints (1200/720/520) to these, or keep that `@media` block in code.

## Responsive & components
- **Make ONE component responsive** instead of shipping a mobile-duplicate: native breakpoint styles + (when a shared embed needs a mobile-only tweak) edit the shared CSS/JS embed once, scoping any JS with `matchMedia` so other breakpoints stay byte-for-byte identical. тЖТ worked-examples E/F. Component **variants** (`data_component_variants_tool`) give per-component style overrides for non-breakpoint variations.
- **Author a reusable component headlessly:** build one instance (`data_whtml_builder`) тЖТ `data_component_tool transform_element_to_component` тЖТ define props (`data_component_props_tool create_prop`) тЖТ **bind each prop to its element** (`data_element_settings_tool set_settings` with `binding {source_type:'prop'}`) тЖТ reuse via `insert_component_instance` + `set_component_instance_prop_values` (a `variant` value selects a variant per instance). Reuse a component only where the pattern **visually matches its default**; where the design diverges, build bespoke elements and say why тАФ don't force the component.
- **Shared-class / shared-embed blast radius:** class names are global site-wide and embeds are frequently shared components тАФ one `update_style` or embed edit hits every instance. Check reach with `data_component_tool` `includeInstanceCount`; usually a feature (consistent everywhere), but say so to the user. тЖТ Pitfall 18.
- **Removing a redundant component:** no bulk instance-delete exists and `unregister_component` errors while the component is in use тЖТ hand the user the Designer **right-click тЖТ Delete Component** (removes definition + every instance atomically, undoable). тЖТ Pitfall 20.

## Custom code via the API тАФ freeform head/footer + HTML embeds
- **Site/Page freeform head + footer:** `data_scripts_tool` `get_site_freeform_code`/`set_site_freeform_code`, `get_page_freeform_code`/`set_page_freeform_code` (location `head`|`footer`). The setters **REPLACE the whole block** (no append) тАФ a blind `set_site_freeform_code` nukes existing schema/analytics/reveal code, so always read тЖТ modify тЖТ write back.
- **HTML embeds** (`<style>`/`<script>` inside a component or page): they're `HtmlEmbed` elements тАФ read/write via `data_element_settings_tool` `get_settings`/`set_settings`, key **`code`** (`static_text.value` = the whole `<style>тАж</style>`/`<script>тАж</script>` string). Find them with `data_element_tool query_elements` `element_filter.type:"HtmlEmbed"` (`scope_component_id` for component-nested embeds). Still snapshot the original block to a local file before a big rewrite тАФ and because both APIs make you re-send the WHOLE block, extract the original from the live page mechanically and diff it back after publishing (тЖТ Pitfall 33).
- **Registered site scripts are the wrong default.** `register_inline_script` caps source at 2000 chars, is invisible to the site owner (not in Site Settings тЖТ Custom Code), needs a publish per change, and can never be deleted (`delete_registered_script` 400s). Put site-wide JS in an `HtmlEmbed` inside the component that owns it. тЖТ Pitfall 43.

## Assets & fonts (headless upload тАФ a faithful rebuild needs the real images + fonts)
- **Images:** `data_assets_tool create_asset` (file name + MD5 of the bytes) тЖТ POST the bytes to the returned presigned-S3 form тЖТ `set_image_asset` on the element by asset ID; alt text via `update_asset`. Upload first, bind by ID (whtml won't bind a raw CDN URL). Big JP/PNG тЖТ optional `compress_assets` to webp/avif (replaces the file in place тАФ confirm first).
- **Self-hosted fonts:** `data_fonts_tool create_font` (same presigned-S3 + MD5 flow). A Google Fonts `css2` request returns ONE variable-font file per family (pass the `wght` axis range in `axes`), so a whole family is ~1-3 uploads тАФ self-hosted is GDPR-friendlier than the Google CDN. Fallback: the user adds the family in Site Settings тЖТ Fonts by hand.

## Third-party inline embeds (Calendly & co)
A scheduler/map/form "inline widget" is an `<iframe>` you theme with the vendor's URL params (Calendly: `background_color` / `text_color` / `primary_color` / `hide_gdpr_banner`; the base profile URL renders the full event-type **catalog**, an event URL a single scheduler). Put the vendor's `<div>` + loader `<script>` in ONE `HtmlEmbed` (set its `code` via `data_element_settings_tool`), wrapped in a native styled frame (`overflow:hidden` for rounded corners; background bound to a brand token so the widget blends seamlessly). It loads fine under a visibility-gated / `autoAlpha:0` parent (the iframe still fetches).

**Registered scripts, iOS paint & big-rewrite safety:** 43 **`register_inline_script` is a maintenance trap** тАФ 2000-char cap on the source, **invisible to the site owner** (not in Site Settings тЖТ Custom Code, not in the Designer), applied changes need a publish, and `delete_registered_script` always **400s** so the registry is append-only тЖТ ship site-wide JS as an `HtmlEmbed` in the component that owns the behaviour, filed the way the owner already files things (`CSS` embed / `JS` embed) ┬╖ 44 **iOS-only blank panel** тАФ a scrollable, absolutely-positioned panel inside a `position:fixed` sheet isn't painted on real iOS (invisible to Playwright/WebKit + Chrome device mode; transforms are NOT the cause) тЖТ **portal** the real node to `<body>` as `position:fixed` sized from stable neighbours, restore it exactly on close, and pay back what the owning JS can no longer reach (single-active-state, mirrored drag transform); diagnose on-device with a `?diag=1` overlay that prints state as text ┬╖ 45 **a page serves TWO stylesheets** (`тАжshared.min.css` + `тАжopt.min.css`) and `cssRules` throws cross-origin from JS тЖТ fetch every `<link rel=stylesheet>` and grep the whitespace-stripped concatenation; `getComputedStyle` stays the ground truth.

## Token-efficient workflow
- **One `data_whtml_builder` per section** > many element/style calls.
- **Read the page ID from the live DOM** (`document.documentElement.getAttribute('data-wf-page')`) instead of `list_pages`.
- `data_pages_tool list_pages` returns huge payloads (auto-saved to a file) тЖТ **jq/grep the saved file**; paginate with `limit`/`offset` (`total` is in `pagination`). Same for any 50k+ tool response тАФ slice the file, never re-read it into context.
- **Query styles narrowly:** `data_style_tool get_styles` by `name_path`; add `include_breakpoints`/`include_properties` only when you need them (breakpoint queries are the heavy ones).
- **Verify with measurements, not screenshots:** JS `getComputedStyle` / `getBoundingClientRect` / CSSOM in the browser, or `element_snapshot_tool` (single-element PNG). Full-page screenshot only for the final human-facing proof тАФ and note the in-app browser captures from **scroll-position 0 only** and Webflow **lazy-loads images**, so a full-page shot silently misses below-the-fold visuals; force `loading=eager` + scroll the page through first, or take the proof shot with a headless engine. The pane also does **not fire IntersectionObserver or scroll-driven reveals** (non-painting tab), so scroll-in animations read as "stuck hidden" there тАФ verify those in a headless WebKit run, not the pane. тЖТ Pitfalls 31-32.
- **Batch** style/element writes via the `actions[]` array; **edit a shared embed once** instead of per page.

## Verify on staging (browser, not guesswork)
1. Publish to staging (below).
2. **Propagation lags** тАФ re-navigate once or twice; confirm the new bundle is live by checking the page-CSS hash changed (`site.webflow.<pageId>.<hash>.opt.min.css`) or by fetching the CSS and grepping for your rule тАФ **fetch BOTH stylesheets** (`shared` + `opt`) and strip whitespace, since a page links two and `cssRules` throws cross-origin from JS тАФ THEN measure. тЖТ Pitfalls 21, 45.
3. Assert computed styles / rects + `read_console_messages` (expect zero errors) across the REAL scenarios (announcement banner on/off, each breakpoint, menu open, sticky at rest & scrolled). Screenshot last.
4. Chromium can't reproduce Safari bugs тЖТ flag any fixed-inside-transform / reveal-animation risk for a real-device retest. A **headless WebKit engine** (e.g. Playwright's WebKit) catches many тАФ not all тАФ of those render quirks the in-app Chromium browser can't, and works without a physical device. тЖТ Pitfall 19.

## Build / migration playbook
1. **Recon:** `data_agent_instructions_tool` (site rules) ┬╖ `data_variable_tool get_variables` (nameтЖТID map; social.plus pre-mapped in references/variable-ids.md) ┬╖ `data_style_tool get_styles name_path` (existing classes) ┬╖ read the target page's live DOM. Back up any freeform block you'll rewrite (`data_scripts_tool get_site_freeform_code` / `get_page_freeform_code` тЖТ local file). For a **multi-page site or a Claude Design `.dc.html` canvas**, split frames by `data-screen-label` and lock a shared **class contract** (shared `bb-`-style names + per-page prefixes) up front, so parallel section-builds don't collide on naming тАФ Anti-pattern A10.
2. **Build** the section with `data_whtml_builder` (element-by-element only when you need fine control).
3. **Categorize** each remaining rule: native-eligible тЖТ `update_style`/`create_style`; code-only тЖТ freeform/embed; hybrid section-root тЖТ `--var:` in code, properties native.
4. **Bind variables** (`variable_as_value`) wherever a value matches a brand token; always keep a literal fallback in code.
5. **Rebuild JS-injected static content** as native elements; keep JS for behavior only, wrapped in `Webflow.push()`.
6. **Trim the code** тАФ delete every rule now native (cascade conflicts are the #1 killer). Write embeds/freeform back via the API (`set_site_freeform_code` / `set_page_freeform_code` / embed key `code`) тАФ the setters replace the whole block, so read-then-modify-then-write (Pitfall 33).
7. **Publish to staging + verify** (above). Iterate. Production ONLY on explicit go-ahead.

## Publish
`data_sites_tool publish_site` тЖТ `publishToWebflowSubdomain: true`, `customDomains: []` = staging (`*.webflow.io`) only. Omit both and it **400s** тАФ *"You must pass at least one valid domain id"*; the endpoint **429s** frequently тЖТ wait ~30-60 s and retry (params validate before the rate check, so a `429` rather than a `400` means your params are right). A **full publish pushes ALL pending changes** to production тАФ do it only on explicit user go-ahead. For isolated iteration, the branch API (`data_pages_tool create_branch тЖТ publish_branch тЖТ merge_branch` with conflict resolution) keeps work off main.

## When NOT to use
Pure Webflow work with no design/prototype input тЖТ generic `webflow-skills:*`. One-off CSS tweaks with no migration. Pure Claude-artifact iteration with no Webflow target. Content writing / CMS edits тЖТ the content skills + `webflow-skills:safe-publish`.
