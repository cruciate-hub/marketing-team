# Social.plus Content Matching Logic

This file defines the heuristics for matching an anchor text (found in a partner's article) to the best social.plus target page. The actual inventory of blog posts and glossary entries is NOT hand-curated here — it is auto-generated by the Cloudflare Worker on every CMS publish and lives at:

- **Blog inventory:** [`website/pages-blog.json`](../../../../website/pages-blog.json) — all published blog posts (slug, URL, title, meta description, heading hierarchy)
- **Glossary inventory:** [`website/pages-glossary.json`](../../../../website/pages-glossary.json) — all published glossary entries (same shape)

Both files use the identical item shape (`url`, `metaTitle`, `metaDescription`, `content`) and are always in sync with the live Webflow CMS. Do not re-maintain a list in this file — it will go stale within days.

## How to use this file in `backlink-placement-finder`

1. Fetch **this file** for the matching logic below.
2. Fetch `website/pages-blog.json` and `website/pages-glossary.json` for the actual inventory.
3. When picking a target for a given anchor, apply the heuristics in this file to the `pages[]` arrays from the two JSON files.

## Base URL patterns

- Blog: `https://www.social.plus/blog/{slug}`
- Glossary: `https://www.social.plus/glossary/{slug}`

The JSON files already contain fully-qualified URLs in each item's `url` field. Use them as-is.

## Matching heuristics

The goal: find the social.plus page that is the **most contextually relevant** to the anchor text found in the partner's article.

### 1. Direct topic match (highest priority)

Search both blog and glossary `pages[].metaTitle` and `pages[].content` (heading hierarchy) for the anchor phrase or its root words.

- Anchor "app engagement strategies" → prefer a blog post whose title or H1/H2 literally contains "app engagement strategies"
- Anchor "in-app community" → prefer a blog or glossary entry whose title contains "in-app community"

### 2. Glossary vs. blog choice

Use anchor type to decide which inventory to prefer:

| Anchor type | Prefer | Example |
|---|---|---|
| Short, definitional (1-2 words, generic concept) | **Glossary** | "user engagement", "activity feed", "chat SDK" → `/glossary/{slug}` |
| Specific, strategic, "how-to" (3+ words or action-oriented) | **Blog** | "app engagement strategies", "building brand loyalty" → `/blog/{slug}` |
| Technical SDK/API references | **Either** — whichever has a stronger content match | "chat API", "social SDK" → glossary for definition, blog for comparison/how-to |
| Industry-specific ("retail social features", "fintech community") | **Blog** | Industry content lives in blog posts |

### 3. Depth match

If the anchor is specific, prefer the blog post that dives deep into that subtopic over a general overview. Example: anchor "app engagement metrics" → `/blog/top-app-engagement-metrics-to-measure-apps-performance` beats `/blog/app-engagement-strategies` (which is broader and may compete for the same keywords).

### 4. Target URL Rating check (when Ahrefs available)

Before proposing a target, run `site-explorer-url-rating-history` (or pull from `pages-by-traffic` filtered to the exact URL) on the proposed social.plus target. If URL Rating < 5, swap to a stronger target. Partners reject low-UR targets — domain DR doesn't help if the specific page has no authority.

### 5. Avoid keyword cannibalization

The target page should not compete with the partner article's primary keyword. Example: if the partner article ranks for "community building", don't link them to `/blog/community-building-strategies` — linking there creates a minor SEO conflict. Pick a glossary entry or a blog post on an adjacent topic instead.

### 6. Avoid competitor-linked articles

If Tier 2 of Step 0 flagged that the partner already links to social.plus competitors (Bettermode, Mighty Networks, Discourse, Tribe, Circle, etc.), check whether the candidate partner article already contains a competitor link before inserting ours. If yes, either pick a different article or propose a swap-pitch to Stefan.

## Freshness

Both JSON files are regenerated automatically by the `socialplus-site-json` Cloudflare Worker on every CMS publish of the corresponding collection. If you believe the data is stale, trigger a manual refresh via `/generate/blog?token=...` or `/generate/glossary?token=...`.

## If a JSON file is missing

If `website/pages-blog.json` or `website/pages-glossary.json` cannot be fetched, surface the failure to Stefan immediately. Do **not** fall back to guessing or to hand-curated lists — this file no longer contains them.
