## Core principle
**Native first, code only for what Webflow can't express.** Native styles are Style-panel-adjustable and publish cleanly; code that duplicates a native rule silently overrides it by cascade order. For the initial build of a whole section, **`data_whtml_builder` in one call** beats rebuilding element-by-element.

## Fastest build path â€” `data_whtml_builder`
One call turns `html` + `css` into a native Webflow section (auto-mapped to classes + breakpoints). Constraints (from the Webflow guide):
- **Single root element.** No `<style>` in `html` â€” CSS goes in the `css` param.
- **No `@keyframes`, no custom `@media`** â€” Webflow breakpoints only (below). Keep keyframes / custom-media in freeform head code.
- Build the section, THEN refine natively (bind variables, split hybrids). Far fewer tokens than N Ă— `data_element_builder` + `create_style`.
- **Quirks to fix right after the call (â†’ Pitfalls 22-24):** write combo-class CSS as the **full chained selector** (`.base.combo{â€¦}`, never `.combo{â€¦}` alone â€” a bare combo selector spawns an orphan global rule + an *empty* combo record); it **drops `target="_blank"`** and leaves `href`/`rel` as raw attributes â†’ re-set the link via `data_element_settings_tool set_settings` key `link`, then `remove_attribute`; and it does **not** bind `<img src>` to the asset library even at the canonical CDN URL â†’ `set_image_asset` by asset ID afterwards. It DOES keep inline `<svg>` as native DOM, plus `<br>`, soft hyphens and curly quotes verbatim â€” no HtmlEmbed detour for those.

## Native vs code decision
**Native** (`data_style_tool` Style record): single-class rules; combo classes (`parent_style_names`); supported pseudos (`:hover :focus :focus-visible :active ::before ::after :nth-child(odd/even) :first/last-child :focus-within â€¦`); responsive breakpoint values; variable bindings (`variable_as_value`).
**Code** (freeform head/footer, or an HTML embed): `@keyframes`, `@property`; descendant (`.a .b`) + attribute (`[data-x]`) selectors; `:nth-child(N)`; custom / feature queries (`prefers-reduced-motion`); `color-mix()` (Designer corrupts it on save â€” keep in code, OR store it as a variable `custom_value` and bind the property to that variable); section-root `--var:` definitions descendants inherit; all JS behavior.

## Breakpoints (authoritative)
`main` = base/desktop (â‰Ą992) Â· `medium` â‰¤991 (`max-width:991px`) Â· `small` â‰¤767 (`max-width:767px`) Â· `tiny` â‰¤479 (`max-width:479px`) Â· up-scaling `large`â‰Ą1280 / `xl`â‰Ą1440 / `xxl`â‰Ą1920. **Styles cascade DOWN** (mainâ†’mediumâ†’smallâ†’tiny): set at `small` and it also applies at `tiny` unless overridden. ## Responsive & components
- **Make ONE component responsive** instead of shipping a mobile-duplicate: native breakpoint styles + (when a shared embed needs a mobile-only tweak) edit the shared CSS/JS embed once, scoping any JS with `matchMedia` so other breakpoints stay byte-for-byte identical. â†’ worked-examples E/F. Component **variants** (`data_component_variants_tool`) give per-component style overrides for non-breakpoint variations.
- **Author a reusable component headlessly:** build one instance (`data_whtml_builder`) â†’ `data_component_tool transform_element_to_component` â†’ define props (`data_component_props_tool create_prop`) â†’ **bind each prop to its element** (`data_element_settings_tool set_settings` with `binding {source_type:'prop'}`) â†’ reuse via `insert_component_instance` + `set_component_instance_prop_values` (a `variant` value selects a variant per instance). Reuse a component only where the pattern **visually matches its default**; where the design diverges, build bespoke elements and say why â€” don't force the component.
- **Shared-class / shared-embed blast radius:** class names are global site-wide and embeds are frequently shared components â€” one `update_style` or embed edit hits every instance. Check reach with `data_component_tool` `includeInstanceCount`; usually a feature (consistent everywhere), but say so to the user. â†’ Pitfall 18.
- **Removing a redundant component:** no bulk instance-delete exists and `unregister_component` errors while the component is in use â†’ hand the user the Designer **right-click â†’ Delete Component** (removes definition + every instance atomically, undoable). â†’ Pitfall 20.

## Custom code is API read/write now (reverses the old paste-by-hand advice)
- **Site/Page freeform head + footer:** `data_scripts_tool` `get_site_freeform_code`/`set_site_freeform_code`, `get_page_freeform_code`/`set_page_freeform_code` (location `head`|`footer`). Read â†’ edit â†’ write back; no human paste.
- **HTML embeds** (`<style>`/`<script>` inside a component or page): they're `HtmlEmbed` elements â€” read/write via `data_element_settings_tool` `get_settings`/`set_settings`, key **`code`** (`static_text.value` = the whole `<style>â€¦</style>`/`<script>â€¦</script>` string). Find them with `data_element_tool query_elements` `element_filter.type:"HtmlEmbed"` (`scope_component_id` for component-nested embeds). Still snapshot the original block to a local file before a big rewrite â€” and because both APIs make you re-send the WHOLE block, extract the original from the live page mechanically and diff it back after publishing (â†’ Pitfall 33).
- **Registered site scripts are the wrong default.** `register_inline_script` caps source at 2000 chars, is invisible to the site owner (not in Site Settings â†’ Custom Code), needs a publish per change, and can never be deleted (`delete_registered_script` 400s). Put site-wide JS in an `HtmlEmbed` inside the component that owns it. â†’ Pitfall 43.

## Assets & fonts (headless upload â€” a faithful rebuild needs the real images + fonts)
- **Images:** `data_assets_tool create_asset` (file name + MD5 of the bytes) â†’ POST the bytes to the returned presigned-S3 form â†’ `set_image_asset` on the element by asset ID; alt text via `update_asset`. Upload first, bind by ID (whtml won't bind a raw CDN URL). Big JP/PNG â†’ optional `compress_assets` to webp/avif (replaces the file in place â€” confirm first).
- **Self-hosted fonts:** `data_fonts_tool create_font` (same presigned-S3 + MD5 flow). A Google Fonts `css2` request returns ONE variable-font file per family (pass the `wght` axis range in `axes`), so a whole family is ~1-3 uploads â€” self-hosted is GDPR-friendlier than the Google CDN. Fallback: the user adds the family in Site Settings â†’ Fonts by hand.

## Third-party inline embeds (Calendly & co)
A scheduler/map/form "inline widget" is an `<iframe>` you theme with the vendor's URL params (Calendly: `background_color` / `text_color` / `primary_color` / `hide_gdpr_banner`; the base profile URL renders the full event-type **catalog**, an event URL a single scheduler). Put the vendor's `<div>` + loader `<script>` in ONE `HtmlEmbed` (set its `code` via `data_element_settings_tool`), wrapped in a native styled frame (`overflow:hidden` for rounded corners; background bound to a brand token so the widget blends seamlessly). It loads fine under a visibility-gated / `autoAlpha:0` parent (the iframe still fetches).

## Token-efficient workflow
- **One `data_whtml_builder` per section** > many element/style calls.
- **Read the page ID from the live DOM** (`document.documentElement.getAttribute('data-wf-page')`) instead of `list_pages`.
- `data_pages_tool list_pages` returns huge payloads (auto-saved to a file) â†’ **jq/grep the saved file**; paginate with `limit`/`offset` (`total` is in `pagination`). Same for any 50k+ tool response â€” slice the file, never re-read it into context.
- **Query styles narrowly:** `data_style_tool query_styles` by `name_path`; add `include_breakpoints`/`include_properties` only when you need them (breakpoint queries are the heavy ones).
- **Verify with measurements, not screenshots:** JS `getComputedStyle` / `getBoundingClientRect` / CSSOM in the browser, or `element_snapshot_tool` (single-element PNG). Full-page screenshot only for the final human-facing proof â€” and note the in-app browser captures from **scroll-position 0 only** and Webflow **lazy-loads images**, so a full-page shot silently misses below-the-fold visuals; force `loading=eager` + scroll the page through first, or take the proof shot with a headless engine. The pane also does **not fire IntersectionObserver or scroll-driven reveals** (non-painting tab), so scroll-in animations read as "stuck hidden" there â€” verify those in a headless WebKit run, not the pane. â†’ Pitfalls 31-32.
- **Batch** style/element writes via the `actions[]` array; **edit a shared embed once** instead of per page.

## Verify on staging (browser, not guesswork)
1. Publish to staging (below).
2. **Propagation lags** â€” re-navigate once or twice; confirm the new bundle is live by checking the page-CSS hash changed (`site.webflow.<pageId>.<hash>.opt.min.css`) or by fetching the CSS and grepping for your rule â€” **fetch BOTH stylesheets** (`shared` + `opt`) and strip whitespace, since a page links two and `cssRules` throws cross-origin from JS â€” THEN measure. â†’ Pitfalls 21, 45.
3. Assert computed styles / rects + `read_console_messages` (expect zero errors) across the REAL scenarios (announcement banner on/off, each breakpoint, menu open, sticky at rest & scrolled). Screenshot last.
4. Chromium can't reproduce Safari bugs â†’ flag any fixed-inside-transform / reveal-animation risk for a real-device retest. A **headless WebKit engine** (e.g. Playwright's WebKit) catches many â€” not all â€” of those render quirks the in-app Chromium browser can't, and works without a physical device. â†’ Pitfall 19.

## Build / migration playbook
1. **Recon:** `data_agent_instructions_tool` (site rules) Â· `data_variable_tool query_variables` (nameâ†’ID map; social.plus pre-mapped in references/variable-ids.md) Â· `data_style_tool query_styles name_path` (existing classes) Â· read the target page's live DOM. For a **multi-page site or a Claude Design `.dc.html` canvas**, split frames by `data-screen-label` and lock a shared **class contract** (shared `bb-`-style names + per-page prefixes) up front, so parallel section-builds don't collide on naming â€” Anti-pattern A10.
2. **Build** the section with `data_whtml_builder` (element-by-element only when you need fine control).
3. **Categorize** each remaining rule: native-eligible â†’ `update_style`/`create_style`; code-only â†’ freeform/embed; hybrid section-root â†’ `--var:` in code, properties native.
4. **Bind variables** (`variable_as_value`) wherever a value matches a brand token; always keep a literal fallback in code.
5. **Rebuild JS-injected static content** as native elements; keep JS for behavior only, wrapped in `Webflow.push()`.
6. **Trim the code** â€” delete every rule now native (cascade conflicts are the #1 killer). Write embeds/freeform back via the API.
7. **Publish to staging + verify** (above). Iterate. Production ONLY on explicit go-ahead.

## Publish
`data_sites_tool publish_site` â†’ `publishToWebflowSubdomain: true`, `customDomains: []` = staging (`*.webflow.io`) only. Omit both and it **400s** â€” *"You must pass at least one valid domain id"*; the endpoint **429s** frequently â†’ wait ~30-60 s and retry (params validate before the rate check, so a `429` rather than a `400` means your params are right). A **full publish pushes ALL pending changes** to production â€” do it only on explicit user go-ahead. For isolated iteration, the branch API (`data_pages_tool create_branch â†’ publish_branch â†’ merge_branch` with conflict resolution) keeps work off main.
